# 图片缩略图与预览图方案

> 更新时间：2026-03-26
> 方案目标：在保证上传响应速度的前提下，为列表页和详情页提供更合适的图片资源。

## 1. 背景与结论

项目需要同时支持三类图片资源：

- 原图：用于下载、查看原始内容、后续编辑
- 缩略图：用于列表页、卡片流、搜索结果
- 预览图：用于详情页主展示区域

最终决策不是“上传时同步生成三份图”，而是采用“性能优先”的实现策略：

1. 上传主链路优先保证快
2. 列表页优先使用缩略图
3. 预览图允许稍后异步生成
4. 详情页在预览图未就绪时回退到原图

实现边界需要明确：

- `x-file-storage` 用于上传原图，并复用其成熟的缩略图能力
- `x-file-storage` 不作为 WebP 预览图的核心实现方案
- `previewUrl` 由独立的异步预览图生成流程负责

这是当前阶段最合理的选择，因为用户已经明确约束：

- 上传响应必须快
- 预览图可以稍后生成

## 2. 两类衍生图的定位

### 2.1 缩略图 thumbnail

缩略图的定位是“批量浏览资源”。

主要使用场景：

- 首页图片列表
- 空间图片列表
- 搜索结果页
- 任何一次需要同时展示多张图片的地方

核心目标：

- 减少带宽消耗
- 降低浏览器批量解码成本
- 提升首屏加载速度
- 改善滚动加载体验

一句话概括：

缩略图解决的是“多张图同时加载太重”的问题。

### 2.2 预览图 preview

预览图的定位是“网页展示版大图”。

主要使用场景：

- 图片详情页主图
- 大图查看前的默认展示资源

核心目标：

- 在清晰度可接受的前提下降低体积
- 避免详情页直接加载超大原图
- 平衡观感和网页性能

一句话概括：

预览图解决的是“单张原图太大，不适合网页直接展示”的问题。

### 2.3 原图的定位

原图始终保留，作为唯一真实源文件，用于：

- 原图下载
- 高清查看
- 后续再编辑或再处理

注意：

预览图并不是“比原图更清晰”，更准确的说法应该是：

- 原图更完整、更保真
- 预览图更适合网页展示

## 3. 为什么采用性能优先方案

### 3.1 为什么不建议同步生成三份图

如果把原图、缩略图、预览图都放在上传请求里同步生成，会带来几个直接问题：

- 上传接口响应时间显著变长
- WebP 转码会增加 CPU 和 IO 压力
- 高并发上传时更容易造成资源竞争
- 一旦预览图生成失败，容易影响主流程稳定性

这和当前“上传响应必须快”的目标冲突。

### 3.2 为什么列表页优先切缩略图

列表页是这个方案里性能收益最大的地方。

原因：

- 一次可能加载几十张图片
- 性能瓶颈通常来自批量请求、批量解码和批量渲染
- 每张图哪怕只减少几百 KB，总体收益也很明显

所以缩略图不是可选优化，而是优先级最高的优化项。

### 3.3 为什么预览图可以异步生成

详情页是单图场景，压力远小于列表页。

因此，预览图可以不进入主上传链路，而是在上传成功后异步补齐。这样做有几个优点：

- 上传立即成功，用户等待更短
- 预览图生成失败不会影响主功能
- 后端可以做失败重试、补偿任务
- 方案更容易渐进落地

## 4. 总体方案

### 4.1 资源分工

- `url`：原图地址
- `thumbnailUrl`：缩略图地址
- `previewUrl`：预览图地址

页面使用规则：

- 列表页优先使用 `thumbnailUrl`
- 详情页优先使用 `previewUrl`
- `previewUrl` 不存在时，详情页回退到 `url`
- 下载始终使用 `url`

### 4.2 上传流程

建议上传流程如下：

1. 用户上传图片
2. 后端完成原图上传
3. 同步拿到并保存 `url`
4. 同步拿到并保存 `thumbnailUrl`
5. 数据入库时将 `previewUrl` 置空
6. 提交异步任务生成预览图
7. 预览图生成成功后更新 `previewUrl`
8. 若生成失败，保留原图可访问，并记录失败状态或日志

这个流程满足两个目标：

- 主链路快
- 页面展示逐步优化

## 5. 后端设计

### 5.1 数据库字段

当前建议至少新增字段：

```sql
ALTER TABLE picture
    ADD COLUMN previewUrl varchar(512) NULL COMMENT '预览图 url';
```

建议再增加状态字段，便于排障和重试：

```sql
ALTER TABLE picture
    ADD COLUMN previewStatus tinyint NOT NULL DEFAULT 0 COMMENT '预览图状态: 0-pending, 1-processing, 2-ready, 3-failed';
```

这样设计的原因：

- 只有 `previewUrl` 能跑功能
- 但增加 `previewStatus` 后，可观测性和可维护性更强

### 5.2 实体与 VO

需要在 `Picture` 和 `PictureVO` 中补充：

- `previewUrl`
- 可选：`previewStatus`

用途：

- 后端保存状态
- 前端判断是否有预览图可用

### 5.3 上传主链路

上传主链路只做必要工作：

- 原图上传
- 通过 `x-file-storage` 获取或生成缩略图地址
- 图片基础信息入库

不建议在主链路中同步执行：

- WebP 预览图转码
- 额外复杂的图片分析

原因：

- 这些操作都属于可延后优化项
- 不应拖慢上传接口

### 5.4 预览图异步任务

预览图异步任务负责：

1. 获取原图
2. 使用独立图片处理逻辑生成 WebP 预览图
3. 上传预览图到对象存储
4. 更新数据库 `previewUrl`
5. 更新 `previewStatus`

这里要特别说明：

- `x-file-storage` 的缩略图能力可以直接复用
- 但 WebP 预览图不应假设由 `x-file-storage` 原生完成
- 预览图应由单独的图片处理组件负责，例如 `webp-imageio` 一类专用库，或者云厂商原生图片处理能力

状态流建议：

- 新上传：`pending`
- 任务开始：`processing`
- 成功：`ready`
- 失败：`failed`

### 5.5 预览图生成策略

预览图建议使用 WebP，并设置网页展示上限，而不是无脑保留原始大尺寸。

建议原则：

- 控制最大宽高，如 `1600px` 或 `1920px`
- 保持纵横比
- 压缩质量以“网页观感合格”为目标，而不是追求原图级保真

这样做的原因：

- 详情页不需要原图级分辨率
- 只改格式不控尺寸，很多大图依然会很重
- 控制尺寸上限通常比只调压缩质量更稳定

### 5.6 缩略图策略

缩略图建议优先复用现有上传体系已经返回的 `thumbnailUrl` 能力。

现阶段不建议一开始就自建完整缩略图处理链，除非现有能力不满足：

- 尺寸不符合要求
- 裁剪策略不符合要求
- 图片质量不可接受

这样做的理由：

- 现有能力已经通了，复用成本最低
- 当前重点是先跑通前后端闭环
- 过早自建图片处理链会增加复杂度和故障面

补充说明：

- `x-file-storage` 的缩略图能力主要适合 resize 和简单裁剪
- 它不能作为预览图 WebP 压缩方案的直接替代
- 因此本方案对缩略图和预览图采用分治处理，而不是统一依赖同一条能力链

## 6. 前端设计

### 6.1 列表页

列表页应优先使用：

- `picture.thumbnailUrl`

如果缩略图缺失，可回退：

- `picture.url`

理由：

- 保证兼容已有历史数据
- 即使缩略图未生成，也不影响功能可用

### 6.2 详情页

详情页主图应优先使用：

- `picture.previewUrl`

如果预览图不存在，则回退：

- `picture.url`

推荐前端取值逻辑：

```ts
const displayUrl = picture.previewUrl || picture.url
```

下载按钮始终使用：

```ts
const downloadUrl = picture.url
```

### 6.3 是否自动切换

当前阶段建议：

- 详情页首次加载时直接按 `previewUrl || url` 取值
- 不强制做“当前页面自动感知异步任务完成并切换”

原因：

- 自动刷新或实时推送会增加实现复杂度
- 这不是第一阶段的关键收益点
- 用户下一次进入详情页时自然会拿到预览图

如果后续体验上有更高要求，再增加轮询或消息通知。

## 7. 容错与清理

### 7.1 容错原则

预览图失败不能影响主功能。

必须保证：

- 原图上传成功后，业务就算成功
- 预览图失败只影响性能优化，不影响可用性

### 7.2 失败处理

异步任务失败时建议：

- 记录错误日志
- 更新 `previewStatus = failed`
- 支持后台重试或定时补偿

### 7.3 删除清理

删除图片时，需要清理：

- 原图
- 缩略图
- 预览图

这样可以避免对象存储残留无主文件。

## 8. 分阶段落地建议

为了降低改造风险，建议按下面顺序实施：

### 第一阶段

- 前端列表页切换到 `thumbnailUrl`
- 前端详情页增加 `previewUrl || url` 回退逻辑
- 后端增加 `previewUrl` 字段

目标：

- 先把最关键的前端资源选择逻辑跑通

### 第二阶段

- 后端接入异步预览图生成
- 上传后将 `previewUrl` 补齐

目标：

- 在不影响上传速度的前提下完成详情页优化

建议实现方式：

- 优先方案：自建异步预览图生成器，使用专用 WebP 处理库
- 替代方案：如果后续存储平台明确支持原生图片处理，再改为云端实时或异步生成

### 第三阶段

- 增加 `previewStatus`
- 增加失败重试与补偿机制
- 视需要优化缩略图尺寸和裁剪策略

目标：

- 提升可观测性、稳定性和运维能力

## 9. 最终结论

本功能的正确实现方向不是“同步生成越多越完整越好”，而是：

- 原图保证主流程可用
- 缩略图通过 `x-file-storage` 优先服务列表性能
- 预览图通过独立异步流程补齐详情页体验
- 一切以“上传快、页面稳、可渐进演进”为第一原则

这套方案与当前项目现状和业务约束一致，能够在较低实现风险下，优先拿到最有价值的性能收益。
